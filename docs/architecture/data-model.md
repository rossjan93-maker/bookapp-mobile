# Data Model

## Current state

Supabase auth is active. No application data tables have been created yet. The model below is the planned v1 target, refined to be ready for the first Supabase schema pass.

---

## Ownership boundary

**Supabase Auth** owns the user account (id, email, password, session).
**The app** must never duplicate those fields. The app extends the user via a `profiles` table, keyed on `auth.users.id`.

---

## Core entities

### profiles
Application-level user record. One row per authenticated user.

| Field | Type | Notes |
|-------|------|-------|
| id | uuid | PK. Maps to `auth.users.id`. |
| username | text | Display name. Unique. |
| yearly_reading_goal | integer | Optional. |
| created_at | timestamptz | |

Notes:
- `profiles.id` is not generated by the app. It is set to the value of `auth.users.id` on sign-up.
- Row-level security (RLS) should enforce that users can only update their own profile.

---

### books
Canonical book record. Sourced from an external search API (e.g. Open Library). Stored locally once a user adds or saves the book.

| Field | Type | Notes |
|-------|------|-------|
| id | uuid | PK. App-generated. |
| title | text | |
| author | text | |
| cover_url | text | Optional. |
| external_id | text | Source API identifier. Used to deduplicate. |

Notes:
- `external_id` should be unique to avoid storing the same book twice.
- A book row is created the first time any user adds it; subsequent users reference the same row.

---

### user_books
Tracks a specific user's relationship with a specific book.

| Field | Type | Notes |
|-------|------|-------|
| id | uuid | PK. |
| user_id | uuid | FK → profiles.id |
| book_id | uuid | FK → books.id |
| status | enum | `want_to_read`, `reading`, `finished`, `dnf` |
| started_at | timestamptz | Optional. Set when status → reading. |
| finished_at | timestamptz | Optional. Set when status → finished. |

Notes:
- One row per `(user_id, book_id)` pair. Enforce with a unique constraint.
- When a recipient saves a recommendation, a `user_books` row is created with `status = want_to_read`.
- When the recipient starts or finishes the book, `user_books.status` is updated and `recommendations.status` should be updated in sync.

---

### friendships
Tracks friend requests and accepted connections between users.

| Field | Type | Notes |
|-------|------|-------|
| id | uuid | PK. |
| requester_id | uuid | FK → profiles.id. The user who sent the request. |
| addressee_id | uuid | FK → profiles.id. The user who received the request. |
| status | enum | `pending`, `accepted` |
| created_at | timestamptz | |
| updated_at | timestamptz | Set when status changes. |

Notes:
- Store one row per pair (requester → addressee). Do not create a reciprocal row.
- To check if two users are friends, query for an `accepted` row where either `(requester_id, addressee_id)` or `(addressee_id, requester_id)` matches the pair.
- Only `accepted` friendships should allow recommendations to flow between users.

---

### recommendations
First-class object. A direct user-to-user book suggestion.

| Field | Type | Notes |
|-------|------|-------|
| id | uuid | PK. |
| from_user_id | uuid | FK → profiles.id. Recommender. |
| to_user_id | uuid | FK → profiles.id. Recipient. |
| book_id | uuid | FK → books.id |
| user_book_id | uuid | FK → user_books.id. Nullable. Set when recipient saves/starts the book. |
| status | enum | `sent`, `saved`, `started`, `finished`, `ignored`, `dnf` |
| note | text | Optional message from the recommender. |
| created_at | timestamptz | |
| resolved_at | timestamptz | Nullable. Set when status → finished / ignored / dnf. |

Notes:
- `user_book_id` links the recommendation to the recipient's library entry once they act on it.
- `recommendations.status` and `user_books.status` should be kept in sync as the recipient progresses.
- Lifecycle: `sent` → `saved` (recipient adds to library) → `started` → `finished` or `dnf` or `ignored`.

---

### credibility_events
Records when a recommendation results in a finished book. Source of truth for recommender credibility score.

| Field | Type | Notes |
|-------|------|-------|
| id | uuid | PK. |
| recommendation_id | uuid | FK → recommendations.id. Unique — one event per recommendation. |
| from_user_id | uuid | FK → profiles.id. Recommender who earns the point. |
| to_user_id | uuid | FK → profiles.id. Reader who finished the book. |
| book_id | uuid | FK → books.id |
| created_at | timestamptz | |

Notes:
- Insert one row when `recommendations.status` transitions to `finished`.
- Credibility score for a user = count of `credibility_events` rows where `from_user_id = user.id`.
- `recommendation_id` should be unique to prevent double-counting.

---

### activity_events
Lightweight feed table. Records key events for display in the home feed.

| Field | Type | Notes |
|-------|------|-------|
| id | uuid | PK. |
| actor_id | uuid | FK → profiles.id. The user who performed the action. |
| event_type | enum | See supported types below. |
| book_id | uuid | FK → books.id. Nullable. |
| recommendation_id | uuid | FK → recommendations.id. Nullable. |
| created_at | timestamptz | |

Supported event types for v1:

| event_type | Meaning |
|------------|---------|
| `recommendation_sent` | A user sent a book recommendation to a friend. |
| `recommendation_saved` | A recipient saved a recommended book to their Want to Read list. |
| `recommendation_started` | A recipient started reading a recommended book. |
| `recommendation_finished` | A recipient finished a recommended book. |
| `book_finished` | A user marked a book as finished (not from a recommendation). |

Notes:
- Only events relevant to friends are shown in the feed. Feed query filters by friendship graph.
- Not every action needs an event row. Only the five types above are in scope for v1.
- Do not use this table as a general-purpose audit log.

---

## Entities explicitly out of scope for v1

- Reviews / ratings
- Comments or discussion threads
- Book clubs
- Public posts or a broad social feed
- Bookstore or library integrations
- Complex gamification tables
