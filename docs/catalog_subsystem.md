## §1 — P0/P0.5: catalog write protection (fail-loud, 14 columns)

- **Catalog write protection (fail-loud, 14 columns):** `_books_protect_identity_columns()` is a `BEFORE UPDATE` trigger that **raises** (SQLSTATE `42501`, message prefix `CATALOG_PROTECTED:`) when a non-service-role caller attempts to modify a protected column. Field classification — *user-mutable* (untouched by trigger): `page_count`. *Immutable post-insert*: `title`, `author`. *Fill-if-empty (NULL → non-NULL only; `''` counts as empty for text)*: `external_id`, `cover_url`, `description`, `isbn`, `isbn13`, `publication_year`, `original_publication_year`, `additional_authors`. *Provider-only fill-empty* (NULL or empty-array allowed → non-empty; overwriting a non-empty value requires service-role): `subjects`, `content_warnings`, `cover_source`, `metadata_confidence`. Both year columns are protected because Goodreads import populates both (`Year Published` → `publication_year`, `Original Publication Year` → `original_publication_year`) and both are catalog-shared. Bypass branch (`auth.uid() IS NULL`) covers no-end-user-JWT contexts: service-role JWTs, direct DB sessions used by migrations / edge functions, and anon (anon is safe because RLS on `books` UPDATE matches 0 rows for non-library-owners before the trigger fires). Empty-string text values count as empty for fill-empty semantics, since Goodreads import + legacy CSV paths can persist `''` instead of `NULL`. Migration `20260509000000_p0_5` replaced the prior silent-revert behaviour from `20260508000000` (P0) because silent reversion hid bugs and made the write contract impossible to reason about — and extended protection from 5 columns to 14 because the previously unguarded columns (`isbn`, `isbn13`, `publication_year`, `original_publication_year`, `additional_authors`, `metadata_confidence`, `cover_source`, `subjects`, `content_warnings`) are shared across users and were freely overwritable by any library owner. Companion identity-column triggers protect `book_source_links.(book_id, source)` and `book_enrichment_cache.external_id` — the natural ON CONFLICT keys — from drift. Two code-side adjustments paired with this migration: (1) `lib/metadataRepair.ts:386` gained the missing `!hasSubjects` gate so the subjects-fill in the same patch as description/page_count fills cannot atomically reject the whole UPDATE; (2) `lib/subjectRepair.ts` tightened its safety guard from `>= 3` to `> 0` so runtime (anon) callers only fill NULL subject lists — sparse-overwrite (1–2 entries → richer) now requires running `scripts/repairSubjectCoverage.ts` with `SUPABASE_SERVICE_ROLE_KEY`. The cover-upgrade branch in `lib/metadataRepair.ts` performs its overwrite in a separate `await` so the expected 403 doesn't poison the legitimate fill-empty patches built in the same loop iteration; the upgrade itself is filed as a follow-up (would need a service-role RPC).

## §2 — P1.5a: catalog provenance (audit trail + quarantine flag)

- **Catalog provenance (P1.5a — audit trail + quarantine flag):** `books.provenance_state` (`'legacy' | 'unverified' | 'verified'`, default `'unverified'`, CHECK-constrained) + `books.provenance_inserted_by` (`uuid → auth.users.id`, NULL for service-role/pre-existing rows). Two BEFORE INSERT triggers fire in alphabetical order: (1) `trg_books_set_provenance` overrides client input on the non-bypass path — `provenance_state := 'unverified'`, `provenance_inserted_by := auth.uid()` — so a malicious client cannot lie about provenance; service-role / no-JWT writes (`auth.uid() IS NULL`) keep full control and may set `'verified'` (this is the chokepoint P1.5b will use); (2) `trg_books_validate_insert` is the P1 guardrail extended with anchored external_id format validation — when provided, it must match `^/works/OL[0-9]+W$` OR `^gb:[A-Za-z0-9_-]+$` OR `^gb_[A-Za-z0-9_-]+$`. NULL external_id remains allowed (Goodreads import + manual fallback depend on it); blank rejected. Migration `20260511000000_p1_5a_provenance.sql` backfilled all pre-existing rows to `'legacy'` (honest grandfathering — we do **not** mark them `'verified'`). Companion client change: `lib/goodreadsExecutor.ts` title+author dedup is filtered to `provenance_state IN ('verified','legacy') OR provenance_inserted_by = ${userId}` so a malicious manual entry from user A cannot be absorbed as the canonical row for user B's import. **gb_ vs gb: inconsistency is accepted** in P1.5a (recommender/scan/save-from-rec use `gb:`, onboarding `RecEntryScreen` uses `gb_`); both are allowlisted, normalization deferred to P1.5b. **P1.5b backlog**: server-side OL/GB lookup (edge function or `pg_net`); trusted `upsert_book_from_provider` RPC; verification job that flips `'unverified' → 'verified'` after provider re-check; cross-user surface filtering (search/recs hide `'unverified'`); optional split of manual entries into `user_book_custom_entries`; gb_/gb: prefix normalization.

## §3 — P1.5b-1: catalog verification reconciler (server-side trusted ingest)

- **Catalog verification reconciler (P1.5b-1 — server-side trusted ingest):** Edge Function `supabase/functions/verify-books-batch/` (entrypoint `index.ts`, shared providers under `_shared/providers/{types,rateLimiter,openLibrary,googleBooks,cache,lookupLogger}.ts`) drains `books` rows where `provenance_state IN ('unverified','legacy') AND verification_attempt_count < 5` outside their backoff window, looks them up via Open Library then Google Books, fills empty canonical fields, and flips the row to `'verified'` with `provider_verified_at` + `canonical_provider` set. Migration `20260512000000_p1_5b_1` adds 5 columns to `books` (`last_verification_attempt_at`, `provider_verified_at`, `verification_attempt_count`, `last_verification_error` w/ CHECK length≤500, `canonical_provider` w/ CHECK in `{'open_library','google_books'}`), the `provider_lookup_log` audit table (90-day retention via `purge_provider_lookup_log()`), and three RPCs `verify_books_batch_{acquire,release,force_release}_lock(p_actor text)`. **Lock model:** row-presence in `verify_books_batch_lock` (NOT `pg_try_advisory_lock` — PostgREST opens a fresh DB session per HTTP request and a session-scoped advisory lock would auto-release after acquire returns); stale-lock takeover at `>30 min` so a crashed Edge invocation can't permanently jam the cron; **actor-aware release** — passing actor B to `release_lock` when the lock is held by A returns `false` AND leaves the row bound to A (closes the cross-actor regression caught in review round 4); zero-arg overload of `release_lock()` was deliberately dropped (proven gone via PostgREST `PGRST202`). **State machine on success:** sets `provenance_state='verified'`, `provider_verified_at=now()`, `last_verification_attempt_at=now()`, `canonical_provider`, leaves `verification_attempt_count=0` and `last_verification_error=NULL` (success path does NOT increment — only failures do; first-attempt success therefore keeps the counter at 0 by design). `mergeFields` blocks writes to `title`/`author` even from the reconciler — those are immutable post-insert per the catalog-protection contract; the reconciler only fills empty fields elsewhere. **Auth:** service-role-only via JWT exact-match against `READSTACK_SERVICE_ROLE_KEY ?? SUPABASE_SERVICE_ROLE_KEY`. The aliased name exists because the Supabase CLI **rejects `secrets set` for any name starting with `SUPABASE_`** (reserved prefix); the runtime injects its own `SUPABASE_SERVICE_ROLE_KEY` so the fallback covers fresh deploys. If neither resolves the function returns HTTP **500** `{"ok":false,"error":"missing_service_role_key"}` (deliberately distinct from 401 so the failure points at deploy, not at request headers). Anon EXECUTE on all 4 RPCs is revoked; anon SELECT on both new tables is grant-denied at the PostgREST layer (RLS-only-no-policy plus `revoke all from anon, authenticated` produces 401/42501 at the table-grant level before RLS even runs — strictly stronger than RLS-returns-empty). **Operational state:** deployed via `supabase functions deploy verify-books-batch --project-ref <ref> --use-api`; first smoke-test (`?limit=5`) returned `processed:5, verified:5, failed:0, cache_hits:0, latency_ms:2230`, all 5 books fully populated, all 5 log rows `status='success'` (4 `works_key` + 1 `isbn13` fallback), lock table empty post-run. Two cron triggers active: (1) `verify-books-batch-hourly` at `0 * * * *` POST `?limit=200` (week-1 drain pace; reduce to daily at `0 3 * * *` after backlog clears); (2) `purge-provider-lookup-log-daily` at `15 3 * * *` SQL `select public.purge_provider_lookup_log();`. **Verification harness:** `/tmp/p1_5b_1_verify.mjs` — 62 PASS / 0 FAIL across 10 groups; rerun after any reconciler-touching change. Companion harness `/tmp/p1_5b_1_1_verify.mjs` (8 sections incl. invariants 4d-4f) is the regression check for any future reconciler edit. **P1.5b-2 shipped (2026-05-09):** cross-user surface filtering for the three highest-leverage discovery paths. ## §7 — Gotcha: Books INSERT minimal guardrail

- **Books INSERT minimal guardrail:** `_books_validate_insert` (BEFORE INSERT, fail-loud) requires non-empty trimmed `title` AND `author` for non-service-role inserts. If `external_id` is provided it must be non-empty after trim (NULL still allowed because add-book ISBN-fallback and `goodreadsExecutor` legitimately INSERT with `external_id` NULL — populated later by `metadataRepair`). Service-role bypass via `auth.uid() IS NULL` (consistent w/ P0.5). The deeper concern — first-write poisoning of plausible-but-bad rows for valid-looking external_ids — is **deferred to P1.5** (trusted catalog ingestion / provider validation / low-confidence quarantine).

## §8 — Gotcha: Reconciler service-role key

- **Reconciler service-role key:** the Edge Function resolves `READSTACK_SERVICE_ROLE_KEY` first then falls back to `SUPABASE_SERVICE_ROLE_KEY`. The aliased name is mandatory because `supabase secrets set SUPABASE_…` is rejected by the CLI (reserved prefix). Set with `supabase secrets set READSTACK_SERVICE_ROLE_KEY="$SUPABASE_SERVICE_ROLE_KEY" --project-ref <ref>`. Smoke-testing with `Authorization: Bearer $SUPABASE_SERVICE_ROLE_KEY` from a shell still works because both decode to the same JWT. **Never** log the value — the function logs `{readstack_present, fallback_present, resolved_present, resolved_length}` only; respect that contract in any future edit.

## §9 — Gotcha: Reconciler `verification_attempt_count` semantics

- **Reconciler `verification_attempt_count` semantics:** the success path leaves `verification_attempt_count` unchanged (only failures increment it). A row showing `provenance_state='verified'` AND `verification_attempt_count=0` means "verified on first attempt", NOT "never attempted" — distinguish via `provider_verified_at IS NOT NULL`. The `WHERE` clause that selects candidates uses `verification_attempt_count < 5` and a state filter, so verified rows are excluded regardless of counter value. **P1.5b-1.1 addendum:** the terminal-classification path (see next gotcha) also sets `verification_attempt_count = 5` directly without making any provider call. So `count = 5` no longer implies "5 failed provider attempts" — distinguish via `last_verification_error`: one of `placeholder_manual_entry` / `unsupported_external_id_scheme` / `missing_supported_identifier` means terminal-classified (zero provider calls; check `provider_lookup_log` to confirm), anything else means exhausted-retry.

## §10 — Gotcha: Reconciler terminal classification (P1.5b-1.1)

- **Reconciler terminal classification (P1.5b-1.1):** before any provider attempt, `classifyTerminal(row)` in `verify-books-batch/index.ts` checks whether the row has any actionable verification path. If it doesn't (no ISBN/ISBN13 AND external_id is either NULL, `/works/other_<slug>`, or an opaque non-allowlisted scheme like `goodreads:NNN`), the reconciler skips provider calls entirely, sets `verification_attempt_count = MAX_ATTEMPTS (5)`, and writes one of three diagnostic strings to `last_verification_error`: `placeholder_manual_entry` / `unsupported_external_id_scheme` / `missing_supported_identifier`. Provenance_state is **intentionally left unchanged** — terminal rows stay `'legacy'` (or whatever they were); F4 does NOT mark anything verified. The `onboarding_isbn_<10|13 digits>` scheme is recognized as actionable via F1's `parseOnboardingIsbn()` synth-ISBN injection, so those rows take the normal resolver path. Any new external_id scheme added to the allowlist must also be added to `classifyTerminal()`'s "non-terminal" branch list, or every row using it will be incorrectly terminated. See `docs/p1_5b_1_reconciler_runbook.md` §8 for the full error-string glossary.

## §11 — Gotcha: Reconciler lock primitive

- **Reconciler lock primitive:** never replace the row-presence lock in `verify_books_batch_lock` with `pg_try_advisory_lock` — PostgREST opens a fresh DB session per HTTP request and a session-scoped advisory lock would auto-release after `acquire` returned, eliminating the overlap protection. Always pass an actor string to `verify_books_batch_release_lock(text)`; the zero-arg overload was deliberately dropped and PostgREST will return `PGRST202` if anything calls it bare. Stale-lock takeover threshold is 30 minutes — if a hung run lasts longer the next cron tick will steal the lock; if you need to clear it sooner call `select public.verify_books_batch_force_release_lock();` from a service-role SQL session.

## §12 — Gotcha: Reconciler `mergeFields` invariant

- **Reconciler `mergeFields` invariant:** the merge step in `verify-books-batch` blocks writes to `title` and `author` even from the reconciler — those columns are immutable post-insert per the P0.5 catalog-protection trigger and the reconciler must respect that contract. Any new "fill empty" field added to the reconciler must be paired with a check that the catalog-protection trigger classifies it as fill-empty (NULL → non-NULL allowed); patching a protected column would 403 the entire UPDATE atomically per the multi-column gotcha below.

## §13 — Gotcha: Multi-column `books` UPDATE

- **Multi-column `books` UPDATE:** the catalog-protection trigger is fail-loud and atomic — a single rejected column rejects the entire patch. Any new code that PATCHes `books` from a non-service-role context must isolate potentially-protected writes (e.g. cover upgrades that overwrite a non-empty `cover_url`) into their own `await` so the expected 403 doesn't poison legitimate fill-empty patches built in the same loop iteration. See `lib/metadataRepair.ts` cover-upgrade branch for the established pattern.
