Architecture requirement:
Implement or validate a single recommendation-config identity concept that can be carried by all three deck-state stores and compared on read.

Target behavior:
- A recommendation deck built under config/hash A should not be reused when current config/hash is B.
- On mismatch, the affected store should invalidate itself and force the existing rebuild path rather than returning stale books.
- This must apply to:
  1. persisted payload restore
  2. in-memory recSession reuse
  3. recQueue reuse / append-vs-replace behavior

Implementation requirements:

1. Preflight: inspect current fingerprint/hash state
Before changing code, report internally in your implementation reasoning:
- what fingerprint/hash already exists today, if any
- which preference fields or recommendation-relevant inputs it includes
- where it is currently used
- where it is not used
- how it can serve as the initial P0B recommendation-config identity until RecRequest.configHash arrives in P1

If there is no viable current fingerprint, stop and propose the smallest architecture-compatible alternative.

2. Create centralized validity helper
Add lib/recValidity.ts or the smallest equivalent central helper.

It should own:
- comparison logic for “deck state is valid for current recommendation config”
- consistent mismatch behavior
- a reusable return shape or helper that each store can call

The purpose is to avoid three stores each inventing their own mismatch rules.

3. Persist/store config identity across all deck-state layers
Extend:
- recPayloadCache
- recSession
- recQueue

so each deck state knows which recommendation-config fingerprint/hash it belongs to.

Requirements:
- persisted payload includes the hash/fingerprint
- in-memory session includes the hash/fingerprint
- queue state includes the hash/fingerprint or is otherwise centrally tied to a validity identity in a way that prevents stale queue reuse

4. Validate on every relevant read/reuse path
Introduce validity checks at the places where stale decks can currently survive:

A. recPayloadCache restore path
- if stored hash ≠ current hash, reject/clear payload and force rebuild

B. recSession reuse path
- if session hash ≠ current hash, treat as absent and invalidate it

C. recQueue reuse path
- if queue hash ≠ current hash, clear queue before any append/reuse logic can preserve stale head cards

This must specifically prevent the prior stale-deck failure mode:
- pipeline rebuilds fresh entries
- old queue remains non-empty
- new entries append to tail
- stale head remains visible
