## Scope

### 1. Emit retrieval-phase contributions

For each candidate with `_retrieval_reasons[]`, create one retrieval-phase contribution per retrieval reason.

Use the contribution/provenance model introduced in P3A-1.

Each retrieval contribution should include:
- `phase: 'retrieval'`
- source/kind appropriate to the retrieval reason
- the raw retrieval reason
- branch/source evidence when available
- explanation/display eligibility defaulted conservatively
- no score value unless the model already supports value-less retrieval contributions cleanly

Do not infer scoring causality from retrieval alone.

Retrieval contribution means:
> “Why was this candidate considered?”

It does not yet mean:
> “Why did this candidate rank highly?”

### 2. Attach contributions to scored candidates

Carry the retrieval contributions forward into the `ScoredBook` or equivalent scored candidate shape.

Requirements:
- no score math changes
- no rank changes
- no composition changes
- no RecCard changes
- no explanation changes
- no reservation AND-gate changes
- no behavior changes visible to users

### 3. Preserve legacy fields

Keep:
- `_retrieval_reason`
- `_retrieval_reasons`
- `book.reasons[]`
- `_score_breakdown`
- `audit_flags`

Do not delete or replace legacy fields in this batch.

### 4. Cache-hit coherence

P3A-1 notes say cache-hit/restored candidates initialize `_retrieval_reasons` from the singular cached retrieval reason.

For P3A-2, make sure retrieval contributions are also coherent on:
- live retrieval candidates,
- restored/cache-hit candidates,
- candidates that merge duplicate retrieval reasons within the same pipeline run.

Do not bump cache version unless absolutely necessary. If you think a cache version bump is required, stop and explain why before implementing it.

### 5. Validator / fixture

Add or extend a validator to prove retrieval contributions are correct.

The validator should assert:
1. a candidate with two `_retrieval_reasons` receives two retrieval-phase contributions,
2. `_retrieval_reason` remains equal to `_retrieval_reasons[0]`,
3. singular/dominant provenance remains stable,
4. cache-restored candidates produce coherent retrieval contributions,
5. retrieval contributions do not alter ordering/ranking,
6. stated-branch priority remains intact,
7. P2 reservation compatibility remains intact.

Use deterministic fixtures. Avoid live API dependency where possible.
