Task type:
Implementation batch with architecture validation first.

Batch:
P1 — Control-Plane Signal Contract + Nonzero Stated-Preference Influence.

Goal:
Implement the next major phase of the locked Recommendation Architecture Blueprint:

1. Introduce a first-class RecRequest / control-plane contract for recommendation generation.
2. Represent recommendation-relevant user inputs as typed signal classes with provenance.
3. Introduce BuildCause propagation at the recommendation request level.
4. Remove the current trust failure where explicit Reading Taste preferences can drop to zero influence for tier-2+ users.
5. Make soft avoid_genres remain meaningful in ranking across all user tiers.

This is the first phase where the recommender begins to truly honor explicit user taste edits for high-signal users.

Strategic context:
Already shipped:
- P0A — Canonical Genre Taxonomy Integrity ✅
- P0A.1 — Retrieval-Side Taxonomy Fold-In ✅
- P0B/P0B.1 — Full Deck Validity / Config Invalidation ✅

Locked architecture direction:
- Free/core Recommendation Control Plane first.
- Paid Semantic Intelligence Plane later.
- P1 establishes the signal/request layer the rest of the Control Plane depends on.
- P2 will later build branch-based retrieval responsiveness.
- P3 will later build contribution-grounded scoring/explanations.

Important expectation:
P1 should make explicit Reading Taste preferences materially influence scoring/ranking at all tiers, including tier 2+.
P1 does NOT yet guarantee a visibly different top-4 deck after radical preference edits in every case — that top-slate guarantee depends on P2 retrieval branch planning.
But P1 must make “explicit prefs have zero effect for high-signal users” impossible.

User failure this addresses:
A user changed Reading Taste from Sci-Fi to History/Biography and the recommendation engine rebuilt correctly but effectively ignored the new explicit preference because:
- favorite_genres blending was tier-gated to tier <= 1
- avoid_genres were also tier-gated
- explicit prefs could therefore have zero effect for tier-2+ users

P1 must close that trust failure in the ranking/control layer.

IMPLEMENTATION REQUIREMENTS
────────────────────────────────

1. Introduce RecRequest v1
Create lib/recRequest.ts.

It should define a typed request object broadly aligned with the locked spec, but only implement fields needed now.

Target shape, adapt if necessary after preflight:

RecRequest {
  userId: string
  signals: {
    statedTaste: {
      favoriteGenres: ...
      readingStyles?: ...
      favoriteAuthors?: ...
    }
    revealedTaste: {
      existing tasteProfile-derived behavioral signals as a read-through reference or typed wrapper
    }
    softAvoids: {
      genres: ...
    }
    currentIntent?: ...
    shortTermFeedback?: ...
  }
  policy: {
    confidenceMode: 'cold_start' | 'thin' | 'high_signal'
    statedPreferenceWeight: number
    statedPreferenceFloor: number
    softAvoidPenalty: number
    decayMultiplier?: number
  }
  build: {
    cause: BuildCause
    configHash?: string | existing P0B hash reference if appropriate
    builtAt: number
    schemaVersion: string
  }
}

Critical:
- Do not invent fields that are completely dead today unless they are needed to preserve the architecture shape.
- It is acceptable for some signal classes to be present as typed placeholders if they are intentionally marked “not yet consumed in P1.”
- Do not build the full future Control Plane in one batch.

2. Introduce BuildCause v1
Define BuildCause at the request layer.

Support at minimum:
- 'session_open'
- 'explicit_preference_edit'

Only propagate causes that have clear present usage today.
If 'manual_refresh', 'intent_apply', etc. are not live yet, define the union only if doing so is harmless and useful for the locked architecture, but do not add behavioral logic for them in P1.

Required current behavior:
- normal For You pipeline build defaults to session_open unless an explicit cause is passed
- Reading Taste save path should make the next build request identifiable as explicit_preference_edit if this can be done cleanly within allowed files
- If clean propagation from app/edit-preferences.tsx to the next RecommendationsFeed build is not viable without new state plumbing, stop and explain the smallest required mechanism rather than half-implementing it

3. Introduce signal provenance v1
Create the smallest viable signal layer.

At minimum:
- stated durable genres from reader_preferences.favorite_genres
- soft avoids from reader_preferences.avoid_genres
- behavioral/revealed profile signals should remain usable from existing tasteProfile fields, but P1 should explicitly separate “stated” from “revealed” at the request/control layer

The goal is not to rewrite tasteProfile now.
The goal is to prevent stated and revealed signals from being anonymous inputs to later logic.

4. Introduce recPolicy.ts
Create lib/recPolicy.ts to centralize initial P1 policy values.

At minimum, policy should own:
- stated preference nonzero floor
- tier/confidence-mode multipliers if needed
- soft avoid penalty / floor
- decay values if decay is implemented in P1

Important:
The architecture spec said numerical values are calibration hypotheses, not architecture truths.
Make that clear in naming/comments.
Do not scatter numbers throughout recommender.ts.

5. Remove zero-effect tier gate for explicit genres/avoids
In lib/tasteProfile.ts and/or the smallest downstream path:
- explicit favorite_genres must continue to affect genre_affinities for tier 2+ users with a nonzero floor
- explicit avoid_genres must continue to affect genre_affinities or downstream ranking with a nonzero negative influence for tier 2+ users
- row-level updated_at may be used for decay if already cleanly available; if decay complicates this batch, stop and propose deferring decay while still shipping the nonzero floor

Acceptance of P1 does not require perfect final weighting.
It requires:
- zeroing by tier is gone
- explicit stated inputs have measurable, deterministic nonzero influence at all tiers
- the values live in policy, not scattered inline

6. Make recommender consume RecRequest
lib/recommender.ts should take or derive the new RecRequest in the smallest sound way.

Options:
- adapt the existing entry point to accept RecRequest directly
- or introduce a compatibility wrapper that builds RecRequest from existing inputs and passes a structured object into the relevant scoring path

Prefer the path that:
- preserves current behavior outside the intended P1 changes
- avoids a full pipeline rewrite
- creates a clean foundation for P2 branch planner

Do not leave RecRequest entirely unused.
P1 must begin shifting the recommender toward request-driven operation.

7. Soft avoid ranking semantics
P1 should ensure soft avoid_genres are materially represented in ranking influence across all tiers.

At minimum:
- their negative influence must not drop to zero for tier 2+
- the policy lives in recPolicy.ts
- the effect is inspectable through deterministic validation

Do not implement retrieval deprioritization yet; that is P2.

8. Deterministic validation
Add scripts/validate_rec_request.ts or an equivalent sound validation script.

It should verify at minimum:

A3-style request delta:
- a synthetic user/request with no stated History preference
- then with History added
- RecRequest.signals.statedTaste changes deterministically
- relevant request/config identity or serialized control input changes if appropriate

A10-style build cause:
- session_open request preserves session_open
- explicit_preference_edit request preserves explicit_preference_edit

P1 floor validation:
- tier 0, tier 1, tier 2, tier 3 fixture snapshots
- stated History favorite produces nonzero positive stated preference effect at every tier
- stated Sci-Fi avoid produces nonzero negative effect at every tier

If scoring-level validation is too coupled for a small script, validate at the tightest honest layer available and report the limitation explicitly.
Do not pretend a test proves ranking if it only proves policy wiring.
