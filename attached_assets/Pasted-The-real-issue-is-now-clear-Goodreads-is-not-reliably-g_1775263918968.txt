The real issue is now clear:

Goodreads is not reliably giving users a usable download/save flow.
Sometimes the export just opens as raw CSV text in the browser.
That means we cannot keep treating “download the file” as the required path.

I do NOT want us to keep trying to force the browser to show a save/download option.
I want us to design around the fact that this may never appear.

==================================================
NEW PRODUCT DECISION
==================================================

We are no longer designing Goodreads import around downloading a file.

We are designing it around two valid acquisition paths:

1. Choose CSV file
2. Paste raw CSV text

“Open Goodreads export page” is only the acquisition step that gets the user to the data.
It is not the actual import mechanism by itself.

==================================================
WHAT THIS MEANS
==================================================

If Goodreads opens the export as raw text in the browser instead of downloading a file:
- that is NOT an edge case
- that is a supported import path

The user should be able to:
- open the Goodreads export page
- copy the raw CSV contents
- return to readstack
- paste the CSV text
- import successfully

That flow must be clear and intentional.

   - if you have a file, choose it
