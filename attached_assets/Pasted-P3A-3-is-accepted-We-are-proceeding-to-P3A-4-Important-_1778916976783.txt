## 1. Create explanation composer

Add:

- `lib/explanations/compose.ts`

or the doc-approved equivalent.

The composer should accept a scored book / contribution bundle and return a structured explanation output.

Suggested output shape:
- primary reason
- secondary reason(s), if supported
- risk / caution line(s), if supported
- descriptive fit line(s), if allowed
- debug/supporting contribution IDs or kinds for validator use

Keep the output internal and deterministic.

## 2. Faithfulness rules

The composer must distinguish:

### A. Causal contribution explanation
Allowed only when there is an eligible positive scoring contribution above its display floor.

Example:
- `stated_taste_fit` with evidence from `stated_favorite:mystery`
- may support language like “based on your mystery preference”

### B. Descriptive fit
Allowed from book metadata, but must not imply personal causation.

Example:
- “A fast-paced thriller”
- is allowed only as descriptive, not “because you love fast-paced thrillers” unless a real contribution supports that.

### C. Risk / caution
Allowed from negative contributions or hygiene/avoidance signals.

Example:
- soft avoid penalty
- metadata/hygiene caution

### D. Quality-only signal
`quality_reliability` / enrichment must not produce personal-taste language.
It may support generic discovery framing only.

## 3. Evidence limitations to enforce

Use the P3A-3 evidence audit as a hard boundary:

- `stated_taste_fit` may cite specific matched stated preference evidence.
- `hygiene_floor` / metadata penalty may cite audit subflags conservatively.
- `behavioral_fit`, `feedback_fit`, and `genre_affinity` are aggregate today and must remain generic unless specific evidence exists.
- `quality_reliability` must not be treated as personal taste evidence.
- `intent_fit`, `novelty_diversity`, and `repetition_suppression` should not be emitted unless supported by actual current contribution fields.

Do not fabricate specificity.

## 4. Display floors

Use the per-kind display floors already placed in `lib/scoring/contributions.ts`.

Requirements:
- floor checks should be deterministic
- negative contributions should not become positive causal explanations
- zero/absent components should emit nothing
- weak aggregate evidence should either produce no explanation or only a generic, non-overclaiming line

## 6. Validator

Add:

- `scripts/validate_explanation_faithfulness.ts`

It should assert:

1. No causal reason is emitted without an eligible above-floor scoring contribution.
2. `stated_taste_fit` can emit a specific reason only when matching stated evidence exists.
3. `quality_reliability` does not emit personal-taste language.
4. `behavioral_fit` / `genre_affinity` stay generic unless specific evidence exists.
5. Negative penalties surface as risk/caution, not positive reasons.
6. Zero/absent components emit no misleading explanation.
7. Retrieval-only contributions cannot produce causal fit explanations.
8. Existing singular `_retrieval_reason` reservation behavior remains unchanged in canonical fixtures.
9. Ranking/order is unchanged.
10. `book.reasons[]` compatibility is preserved or derivable.

Use deterministic fixtures. Avoid live API dependency.
