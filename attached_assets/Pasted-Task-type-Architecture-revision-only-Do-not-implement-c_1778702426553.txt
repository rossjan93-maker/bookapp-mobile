- recPayloadCache

3. Control Plane target architecture
────────────────────────────────

Define the free/core Recommendation Control Plane.

It should include:

A. Canonical taxonomy
- genres
- eventually traits
- aliases
- UI labels
- recommender keys
- retrieval anchors
- zero silent unmapped labels

B. Signal provenance
Distinct signal classes:
- stated durable taste
- revealed behavioral taste
- soft avoids
- hard avoids later
- current next-read intent
- short-term feedback signals

For each:
- source
- confidence
- persistence horizon
- influence on retrieval
- influence on ranking
- explanation eligibility

C. Recommendation request compiler
Propose a structured request/config object compiling:
- durable taste
- inferred taste
- current intent
- avoid constraints
- exploration policy
- confidence profile
- build trigger / invalidation cause

Also define:
- configHash / versioning
- deck/session validity semantics
- why this replaces hidden tier gates and brittle manual clear logic

D. Retrieval architecture
Design a multi-branch retrieval system that ensures:
- revealed-history candidates matter
- stated-preference candidates matter
- current-intent candidates matter
- exploration candidates matter
- soft avoids affect retrieval
- hard avoids can be supported later
- dense users never silently ignore explicit edits

E. Ranking / reranking architecture
Define component scores for:
- behavioral fit
- stated taste fit
- current intent fit
- avoid penalty
- quality/reliability
- novelty/diversity
- repetition suppression

Answer:
- Should explicit preferences always have nonzero influence?
- Should influence decay by time-since-edit, not by tier?
- What creates a visible deck shift after a radical taste edit?
- Should top-slate assembly guarantee some representation from explicit edit-triggered branches when quality-clearing candidates exist?

F. Explanation architecture
Define how explanations become faithful to actual mechanics:
- only cite score contributions that actually mattered
- distinguish stated vs behavioral vs intent-based reasons
- avoid repeated rationales across the slate
- prevent generic copy from misrepresenting why a book surfaced

- narrative structure
- emotional tone

- pacing
- theme clusters

- What should be precomputed/offline vs generated on demand?

B. Reader Intelligence Layer
How should Readstack understand a reader beyond tags and categories?

Consider:
- reader thesis
- taste motifs across loved books
- contrastive analysis of loved vs dismissed books
- latent preferences that span genres
- cross-category patterns such as:

  “institutional stakes with emotional sincerity”
  “big systems colliding with intimate human stories”

Answer:
- What should be deterministic/structured?
- What should be AI-generated?
- How often should a reader thesis update?
- What evidence should it cite?
- How does it avoid overfitting to thin history?

C. Semantic retrieval branch

Examples:
- A reader loves Project Hail Mary, The Martian, Endurance, and Into Thin Air.
- The system identifies:
  competence under pressure, high stakes, operational problem-solving, forward momentum.
- It retrieves books outside obvious genres that share that deeper appeal.

Explain:
- how embeddings/semantic vectors participate
- how semantic retrieval coexists with structured branch retrieval
- how to guard against “interesting but wrong” drift
- how this becomes the source of “surprising but right” recommendations

D. AI-assisted reranking

- how it remains grounded in retrieved candidate evidence

- whether it is premium-only

E. Premium explanation layer
What becomes possible in paid recommendations that is not available in free?

Examples:
- “This is outside your obvious lane, but it shares the exact thing you reward in your favorites…”

Design:
- how the explanation cites actual reader/book evidence

F. Advanced decision sessions

G. Taste evolution insights
Design paid reflective intelligence:
