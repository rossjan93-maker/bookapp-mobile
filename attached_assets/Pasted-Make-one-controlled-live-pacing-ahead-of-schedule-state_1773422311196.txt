Primary product objectives

A. Pacing must account for the current date
Requirements:
1. Revisit pacing logic so it clearly uses the current date and the user’s yearly goal timing.
2. For reading books with enough data, pacing should be able to express:
   - ahead of schedule
   - on pace
   - behind schedule
3. This should work for:
   - page-based pacing when page_count + current_page exist
   - fallback date-based pacing when page data is incomplete
4. Keep the messaging simple and human:
   - “Ahead of pace — you’re winning”
   - “On pace — target Mar 28”
   - “Behind pace — 12 pages/day to catch up”
   or similarly polished language
5. Do not overstate precision when data is weak.

B. Positive reinforcement
Requirements:
1. The app should celebrate when users are ahead.
2. Add a positive pacing state in book detail and any currently-reading surfaces where it makes sense.
3. This should feel motivating, not cheesy.
4. Keep the visual language aligned with the current reading-app style.

C. Live insights refresh
Reader Insights should feel current, not stale.

Requirements:
1. After a user logs page progress, any derived reader insights that depend on progress history should refresh immediately where practical.
2. Profile should not feel like it is waiting for a cold reload to reflect meaningful progress changes.
3. If needed, add a lightweight re-fetch or state refresh pattern after progress updates.
4. Keep this simple and safe — no heavy architecture rewrite.

D. Currently Reading should reflect momentum better
Requirements:
1. Update Currently Reading cards and/or book detail pacing UI so momentum is more legible.
2. If a user is ahead, show that clearly.
3. If a user is behind, show the catch-up pace clearly.
4. If data is weak, keep fallback messaging honest.
