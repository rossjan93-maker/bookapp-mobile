Implement the ordered onboarding stage refactor exactly as proposed.

We are no longer patching the old onboarding/import logic.

I want you to replace the drifting multi-key inference with one authoritative ordered onboarding stage model.

==================================================
OBJECTIVE
==================================================

Make onboarding deterministic and ordered:

1. intro/welcome
2. live walkthrough
3. final setup (dedicated import/upload page)
4. normal app usage

The user must never:
- see final setup as the first onboarding screen
- skip final setup after walkthrough
- refresh and fall into the wrong onboarding stage
- regress unpredictably because multiple keys drift out of sync

==================================================
APPROVED DIAGNOSIS
==================================================

The current model is broken because onboarding stage is inferred from multiple independent keys.

Key visible bug:
`readstack_walkthrough_v1 ?? 'done'` causes the app to treat a missing walkthrough key as if the walkthrough already finished.
That is why import can appear first.

Second visible bug:
welcome completion is stored separately in the database, while walkthrough and final setup live in local keys, so tests can easily drift into invalid combinations.

Third visible bug:
final setup can be lost after walkthrough because the app infers “what stage am I in?” from multiple stale values instead of one authoritative stage.

==================================================
NEW AUTHORITATIVE STATE MODEL
==================================================

Create a new single source of truth key:

`readstack_onboarding_stage_v1`

Allowed values:
- null
- walkthrough
- final_setup
- done

Meaning:
- null = no local onboarding stage set yet
- walkthrough = welcome completed, walkthrough in progress
- final_setup = walkthrough complete, final import/upload step pending
- done = onboarding complete / final setup resolved

Important:
`readstack_walkthrough_v1` stays only for walkthrough sub-step resumption.
It should only matter when stage = `walkthrough`.

Do NOT continue using old import-ob logic as the primary stage driver.
Do NOT infer stage from multiple drifting flags.

==================================================
REQUIRED FLOW
==================================================

Brand new user:
- welcome screen first
- on welcome completion, write onboarding stage = `walkthrough`
- also initialize walkthrough sub-step = `home`
- then enter app shell and start walkthrough

Mid-walkthrough:
- if stage = `walkthrough`, resume walkthrough from `readstack_walkthrough_v1`

When walkthrough completes:
- write onboarding stage = `final_setup`
- route immediately to the dedicated final setup page

While final setup is pending:
- refresh must return the user to the dedicated final setup page
- not back to intro
- not back to bare tabs
- not to the wrong screen

When user dismisses final setup:
- write onboarding stage = `done`
- route into normal app usage

When user taps import:
- write onboarding stage = `done`
- then proceed to import flow
- do not show final onboarding setup again

When import completes successfully:
- keep stage = `done`

Returning/pre-existing users:
- if onboarding stage is null and welcome DB flag says onboarding already completed, do not force onboarding
- they should enter the normal app
