Primary product objectives

A. Preference capture for future taste-fit
Users need a clear way to tell the app what they tend to like.

Requirements:
1. Add a lightweight preference capture/edit experience.
2. It should support at minimum:
   - favorite genres
   - genres they tend to avoid
   - favorite authors
   - preferred pacing/style if useful (examples: fast-paced, reflective, dense, plot-driven, character-driven)
   - optional reading intent / mood preferences if small and clean
3. Keep the UX simple:
   - chips / toggles / compact text entry
   - not a giant survey
4. Preferences should be editable from Profile.
5. Store preferences in the smallest safe schema possible.
6. This is the foundation for future taste-fit, so structure the data cleanly.

B. Profile redesign
Profile needs to do more than show static counts.

Requirements:
1. Add a strong “Currently Reading” section to Profile.
2. This section should feel visually important.
3. Show current books with covers and useful status context.
4. If feasible without new packages, use one of:
   - horizontal card rail
   - stacked cover treatment
   - compact carousel-like scroll
5. Redesign the stats area so these feel more impactful:
   - Friends
   - Books Finished
   - Recs Landed
   - Finished from Recs
6. Use stronger visual hierarchy than the current grid.
7. If helpful, incorporate stacked cover imagery or micro-visuals rather than only plain numbers.
8. Keep Friend Requests and Recommendations Sent accessible and working.

C. Currently Reading should be a first-class surface
Requirements:
1. The app should clearly surface books with status = reading.
2. This should appear prominently on Profile.
3. If useful, also improve Library / Book Detail to support the reading-state experience.
4. For each currently reading book, try to show:
   - cover
   - title
   - author
   - reading status
   - pacing guidance if available
5. Make this section feel like an active reading dashboard, not just another list.

D. Pacing foundation
I want pacing to become a real part of the product.

Requirements:
1. Keep the existing yearly reading goal concept and make it more useful.
2. For books a user starts reading, calculate a pacing target that helps them stay on track for their yearly goal.
3. This should be grounded in the best available information.

Preferred pacing logic:
- If enough page-level information exists:
  - use page count and user reading pace data to estimate pages/day needed
- If page-level information is incomplete:
  - fall back to a simpler per-book cadence / target finish date that still supports the yearly goal

4. Introduce the smallest clean data foundation needed for this.
5. If necessary, add minimal fields such as:
   - page count support
   - current page / reading progress
   - progress update timestamp
   only if truly needed and only with the smallest safe migration

6. If the system does not yet have enough data to estimate a user’s reading speed, handle it gracefully:
   - show a simple target finish date
   - or show a lightweight “pace guidance coming into focus” style message
   - do not fake precision

7. Once a book is in Reading, the user should be able to see pacing guidance somewhere useful:
   - Profile Currently Reading section
   - Book Detail
   - or both
