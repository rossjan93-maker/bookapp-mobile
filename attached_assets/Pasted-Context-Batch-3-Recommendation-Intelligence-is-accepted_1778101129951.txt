PART 2 — Shelf data model

Add a v1 shelf system.

Requirements:
- users can create custom shelves
- users can remove/delete custom shelves
- users can add books to shelves
- users can remove books from shelves
- one book can belong to multiple shelves
- shelves are user-specific
- shelf ordering should be possible later, even if not fully implemented now

Proposed model if nothing better exists:
- `user_shelves`
  - id
  - user_id
  - name
  - created_at
  - updated_at
  - sort_order optional
- `user_shelf_books`
  - id
  - user_id
  - shelf_id
  - user_book_id
  - created_at

Acceptance criteria:
- migration is created
- RLS is correct
- duplicate book-in-shelf entries are prevented
- deleting a shelf removes shelf membership rows but not books

PART 3 — Library shelf UX

Add a simple shelf management experience.

Requirements:
- Library should show default status filters as it does now
- Add a clear “Shelves” entry/section
- Users can create a new shelf
- Users can open a shelf and see books in it
- Users can add books to a shelf through a search/select flow
- Users can remove books from a shelf

Design direction:

- do not make shelves the dominant Library experience yet

Acceptance criteria:
- create shelf
- add book
- view shelf
- remove book
- delete shelf

PART 4 — Want to Read filtering

Improve Want to Read as a saved-book discovery surface.

Problem:
Want to Read is currently just a saved list. It should help users decide what to pick next.

Requirements:
- Want to Read should be expandable/collapsible if needed
- Add filter/search within Want to Read
- Support filtering by available metadata:
  - title
  - author
  - genre/subjects
  - themes
  - mood/tone if available
  - pacing/traits if available
- If metadata is thin, be honest and fall back to title/author/subject matching

Example user need:
“I’m in the mood for a mystery novel — do I have anything saved?”

V1 implementation can be a search/filter input, not full AI chat.

Acceptance criteria:
- user can search within Want to Read
- queries like “mystery”, “romance”, “fantasy”, “short”, “dark”, “light”, “fast paced” return relevant saved books when metadata supports it
- empty states explain whether no books match or metadata is limited

---

PART 5 — Mood / intent matching architecture

Add a lightweight intent parser for Want to Read filtering.

Requirements:
- do not call this “AI” unless there is an actual model call
- use local rules first
- map common intent words to metadata signals

Examples:
- “mystery” → mystery / detective / crime / thriller subjects
- “light” → humor / cozy / romance / low intensity if available
- “dark” → gothic / horror / noir / bleak / psychological
- “fast paced” → pacing trait if available, thriller/action subjects otherwise
- “short” → low page count if page_count exists
- “literary” → literary fiction / prose / character-driven
- “fantasy” → fantasy / magic / epic / romantasy

Acceptance criteria:
- matching logic is centralized in a utility
- easy to extend later
- results explain why they matched when possible

---

PART 6 — Preserve performance

Important:
Library can contain hundreds of books.

Requirements:
- avoid loading unnecessarily huge joined payloads repeatedly
- avoid expensive client-side work on every render
- cache or memoize filtered results where appropriate
- keep scrolling smooth

Acceptance criteria:
- Library remains responsive with 250+ books
- filtering feels instant or near-instant
- no obvious mobile jank
