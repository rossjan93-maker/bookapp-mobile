────────────────────────────────
P2 TARGET BEHAVIOR
────────────────────────────────

After P2, when a user with a dense/high-signal profile explicitly changes Reading Taste to a new lane such as:
- Favorite genres: History, Biography
- Avoid: Sci-Fi

and the next recommendation deck is built with:
BuildCause === 'explicit_preference_edit'

the system should:

1. Preserve quality and hygiene filters.
2. Retrieve some candidates from the newly stated preference lane(s), not only from old behavioral dense lanes.
3. Not allow dense-mode retrieval to bypass stated genres entirely.
4. Carry branch metadata so we know which candidate came from:
   - stated preference branch
   - behavioral/revealed branch
   - author branch
   - intent branch if active
   - fallback/exploration branch if represented
5. Ensure at least one stated-preference branch candidate can surface in the top slate when:
   - BuildCause is explicit_preference_edit
   - at least one stated-branch candidate clears the existing quality/hygiene floor
6. Do not force low-quality books into the top slate merely to satisfy a preference edit.
7. Retain normal behavior for session_open builds without explicit edit cause.

────────────────────────────────
IMPLEMENTATION REQUIREMENTS
────────────────────────────────

1. Introduce retrieval branch planner
Create lib/retrieval/branchPlanner.ts or the cleanest equivalent.

It should:
- take RecRequest + current taste/profile/retrieval inputs needed today
- determine which retrieval branches should run
- determine branch priority/order/quota policy through centralized policy values
- return a retrieval plan or branch outputs in a typed shape
- avoid hidden dense-mode if-statements scattered inside recommender.ts

Initial branch concepts should reflect the locked architecture where feasible:
- statedGenres
- revealedLanes
- revealedAuthors
- intent, only if current intent retrieval semantics already exist and factoring them is clean
- exploration/fallback, only if current logic exists and factoring it is clean

If some future branch is not implementable in this batch, do not fake it. Explicitly defer it.

2. Define branch metadata/types
Create lib/retrieval/types.ts if needed.

Each candidate or candidate set should be able to identify:
- branch kind
- signal provenance class
- originating lane/genre/author/intent where applicable

This metadata is for control-plane correctness now and explanation/scoring faithfulness later.
Do not build P3 explanations yet.

3. Stated-genre retrieval branch
Implement a branch that uses RecRequest.signals.statedTaste.favoriteGenres and the taxonomy-backed retrieval subject anchors from P0A/P0A.1.

Requirements:
- stated genre retrieval must run whenever mapped stated genres exist, including for high-signal/dense users
- the branch must not be silently bypassed by dense mode
- branch output must be separately inspectable in validation
- use current OL retrieval semantics and subject anchors; do not redesign provider queries

4. Replace dense-mode bypass with planner policy
The current dense path can bypass genre affinities/stated preferences entirely.

P2 must:
- preserve the useful revealed-lane/revealed-author behavior
- stop letting that behavior erase stated preference retrieval entirely
- express the balance through planner policy, not a hard bypass

The exact numerical quotas/branch shares are calibration hypotheses.
Place any initial branch policy values in lib/recPolicy.ts if appropriate, or a clearly central branch-policy section in branchPlanner if that is cleaner for P2.
Do not scatter quota numbers through recommender.ts.

5. BuildCause-aware retrieval / slate behavior
Use the P1 BuildCause signal.

For:
BuildCause === 'explicit_preference_edit'

P2 should ensure:
- statedGenres branch receives meaningful retrieval priority
- the final slate assembly has a controlled reservation rule:
  - reserve at least 1 top-slate slot for a stated-preference branch candidate if one clears the quality floor

Important:
- If reservation must occur after scoring, implement the smallest sound slot-assembly hook needed.
- Do not redesign scoring itself.
- Do not force a poor candidate into the slate if none clears the existing quality threshold.
- If top-slate reservation cannot be done cleanly without a broader P3/P4 scoring architecture change, stop and report the smallest alternative.

6. Preserve normal non-edit behavior
For:
BuildCause === 'session_open'

the system should still:
- retrieve from stated preferences as a nonzero branch if the user has them
- but not overcorrect or reserve top-slate slots as aggressively as an explicit edit build

Explicit edit behavior should be visibly responsive.
Normal session behavior should remain balanced.

7. Recommender integration
Refactor only the retrieval orchestration inside lib/recommender.ts as needed to delegate to the planner.

Preserve:
- current scoring path
- current quality/hygiene filters
- current recommendation count contracts
- current queue/session behavior outside natural deck output changes

8. Validation script
Add scripts/validate_retrieval_planner.ts or equivalent.

It must test, at minimum:

A4-style retrieval delta:
- dense/high-signal synthetic request with old behavioral lanes
- explicit favorite History/Biography in RecRequest
- planner includes a statedGenres branch
- planner/retrieval inputs include History/Biography-derived anchors

A5-style top-slate edit response:
- when BuildCause === 'explicit_preference_edit'
- and at least one valid stated-branch candidate exists
- top slate selection reserves/surfaces at least one stated-branch item

A6-style dense-user respect:
- dense/high-signal request still runs revealed lanes/authors
- but no longer excludes stated branch entirely

Session-open contrast:
- session_open runs stated branch without explicit-edit reservation behavior

If full end-to-end candidate retrieval cannot be deterministic without live providers, mock or fixture the branch outputs at the tightest honest layer available. State the validation level clearly. Do not claim an end-to-end deck test if it is planner-level only.
