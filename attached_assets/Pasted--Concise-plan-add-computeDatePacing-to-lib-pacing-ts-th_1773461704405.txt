How Date-Based Pacing State Now Works
The old computePacingNote returned only a string | null. It's now a thin wrapper over a new computeDatePacing function that returns { note: string; state: PacingState } | null.

State is classified honestly using only time:

daysLeft	State	Example note
< 0	'behind'	"Slightly behind — aim to finish soon"
< -14	'behind'	"Behind pace — finish when you can"
≥ 0	'on_pace'	"On pace — target Mar 29"
≥ 0, ≤ 5	'on_pace'	"Finish in 4 days to stay on pace"
'ahead' is never returned from date-based pacing — that can't be determined honestly without knowing how many pages have been read.

In profile.tsx, the fallback branch now populates pacingState from computeDatePacing, which feeds directly into the existing border and chip styling:

on_pace (date-based) → green border, neutral chip — positive momentum
behind (date-based) → amber border, amber chip — flags slippage even before page count is known
No goal / no startedAt → no border, no chip (unchanged)
