Approved: ship the cache-restore retrieval-reason normalization fix.

The investigation is convincing:

- The live OL path works.
- Cached external candidates are restored with `_retrieval_reason` still prefixed, e.g.
  `v5:stated_genre:thriller_mystery`
- Downstream P2 logic expects unprefixed provenance strings such as:
  `stated_genre:thriller_mystery`
- Therefore cache-restored stated candidates become invisible to:
  - P2SURVIVE stated counters
  - P2B.1 retrieval-provenance AND-gate
  - stated reservation eligibility
  - likely other branch-provenance startsWith checks

This is a source-normalization bug on the cache-hit path.

## Required fix

In `lib/recommender.ts`, normalize the restored candidate’s `_retrieval_reason` inside `getCachedExternalCandidates` by stripping the `CACHE_VERSION` prefix before returning the candidate object.

The restored candidate should round-trip to the same semantic provenance string the live OL path uses.

Example:

Before restore:
- DB row: `v5:stated_genre:thriller_mystery`

After restore:
- candidate._retrieval_reason: `stated_genre:thriller_mystery`

## Implementation guidance

- Prefer reusing the same prefix-stripping logic already used elsewhere in the cache-hit trace path.
- If that logic is currently inline only, use the smallest clean shared helper or exact mirrored logic — choose whichever keeps the code clearer without over-expanding scope.
- Fix at the restore source, not by adding downstream special-case checks for `v5:...`.
