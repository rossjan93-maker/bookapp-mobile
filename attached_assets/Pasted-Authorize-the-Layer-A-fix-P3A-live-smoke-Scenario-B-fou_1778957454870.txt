Authorize the Layer A fix.

P3A live smoke Scenario B found a confirmed blocker:
- Fresh quick-taste user stated Nonfiction.
- The app recommended fiction books with the reason:
  “Matches your stated nonfiction preference.”
- Examples:
  - The Fault in Our Stars — John Green
  - The Song of Achilles — Madeline Miller
- Control:
  - Murder in the High Himalaya — Jonathan Green appears to be a valid nonfiction match.

We accept the investigation verdict:
- Composer is not the root cause.
- Projection is not the root cause.
- deriveScoringContributions is not the root cause.
- The issue is upstream in the Book Intelligence Layer:
  `lib/bookTraits.ts::detectGenre`.

Root cause:
- `detectGenre()` uses substring `corpus.includes(...)` matching.
- Broad standalone tokens like `history`, `science`, `psychology`, `philosophy`, `technology`, `sociology` can match fiction-side metadata:
  - “historical fiction”
  - “science fiction”
  - “psychological thriller”
  - “Greek mythology / ancient history”
- `nonfiction` is evaluated before several fiction buckets, so first-match-wins can classify fiction books as nonfiction.
- This produces false `bt.primaryGenre = nonfiction`.
- Then `computeStatedTasteContribution(...)` honestly adds `stated_favorite:nonfiction`.
- Then P3A composer honestly emits “Matches your stated nonfiction preference.”
- Therefore the falsehood begins in `detectGenre`, not in the composer.

Task type:
Targeted Book Intelligence / genre detection fix only.

Goal:
Fix `detectGenre()` so fiction books are not classified as nonfiction due to broad substring matching, while preserving true nonfiction detection.

## 1. Replace substring matching with safer matching

In `lib/bookTraits.ts::detectGenre`:
- Replace `corpus.includes(signal)` style checks with safe word-boundary / token-aware matching.
- Use escaped regex or equivalent tokenization.
- Ensure terms do not match inside larger phrases where they invert meaning.

Examples that must NOT classify as nonfiction:
- `science fiction`
- `historical fiction`
- `psychological thriller`
- `philosophical novel`
- `Greek mythology`
- `young adult fiction`

Use the existing project gotcha as a hard rule:
- subject/content-warning matching should use word-boundary matching, never raw `includes()`.

## 2. Prune overly broad nonfiction signals

Remove or demote standalone nonfiction signals that are too ambiguous in book-subject corpora, especially:
- `science`
- `history`
- `psychology`
- `philosophy`
- `technology`
- `sociology`

Keep unambiguous nonfiction signals such as:
- `nonfiction`
- `non-fiction`
- `self-help`
- `business`
- `economics`
- `politics`
- `true crime`
- clearly nonfiction-specific variants like `popular science`, `science nonfiction`, `history nonfiction`, etc., if already present or easy to add safely.

Do not overfit to only the three smoke books. Fix the matching class.

## 3. Reorder genre detection defensively

Adjust detection order so fiction buckets are not preempted by broad nonfiction matches.

Recommended order:
1. memoir_bio
2. fantasy_scifi
3. thriller_mystery
4. romance
5. horror
6. literary
7. nonfiction
8. general

If you choose a different order, justify it.

## 4. Preserve valid nonfiction controls

Make sure genuine nonfiction still maps correctly.

Positive controls:
- subjects including `nonfiction`
- `true crime`
- `business`
- `self-help`
- `economics`
- `politics`
