Make one targeted “force Goodreads export flow into browser, not app” pass.

Do not change package versions.
Do not add packages unless absolutely necessary.
Do not change schema.
Do not refactor parser, staging, or execution logic.
Do not break the current Goodreads import flow.

Goal:
Fix the mobile Goodreads import experience by preventing users from getting trapped inside the Goodreads app export preview flow.

Observed real-world problem:
When the user taps “Open Goodreads Export Page” from our app, iOS may open the Goodreads app instead of Safari.
Inside the Goodreads app:
- the export opens as an internal preview
- there is no clear Safari download path
- there is no obvious way to get the CSV back into our importer
This is the real blocker.

Desired product behavior:
Guide users into a browser-based flow, not an in-app Goodreads flow.

The ideal user experience is:
1. user opens the flow from our app
2. user is clearly told this must happen in Safari/Chrome, not the Goodreads app
3. if Goodreads opens in the app, the user is told exactly how to get to Safari
4. once the CSV preview opens, the user is told how to save/share it back into a real file they can upload
5. then they return and choose the CSV file

Scope:
Only inspect and edit what is necessary, likely:
- app/import/goodreads.tsx
- maybe one tiny helper if useful
- no unrelated parser/import execution/schema changes

Primary product objectives

A. Add a strong browser-only warning
Requirements:
1. Add a clear high-visibility note near the top of the import screen that says this flow must be completed in Safari or Chrome, not inside the Goodreads app.
2. This should be short, explicit, and hard to miss.
3. It should frame the rest of the flow.

B. Rewrite Step 1 around Safari specifically
Requirements:
1. Step 1 should no longer just say “Open Goodreads in your browser.”
2. It should say that the black button below is meant to open Goodreads in Safari/Chrome.
3. It should explicitly cover the failure case:
   - if the phone opens the Goodreads app instead, use the menu to open in Safari
   - or copy/open the link in Safari manually
4. The wording should be extremely explicit and practical.

C. Make Step 3 reflect the real export behavior
Requirements:
1. Explain that Goodreads may be subtle after tapping Export Library.
2. The app may:
   - quietly add a line under the export button
   - or immediately open the CSV in a preview
3. Reassure the user that both can mean success.

D. Rewrite Step 4 around the actual iPhone preview behavior
Requirements:
1. Remove misleading assumptions about Safari Downloads if that is not the real next step for the observed flow.
2. The new guidance should reflect the working path:
   - if the CSV opens in preview
   - tap the Share button
   - save/share it into Files if available
   - then return to our app and upload that file
3. If the current observation suggests “Open in Safari first, then Share from preview” is the actual best path, reflect that clearly.
4. Optimize for practical success, not theoretical browser behavior.

E. Add an intermediate/browser-safe launch option if helpful
Requirements:
1. If directly opening the Goodreads URL is causing app hijacking, implement the cleanest low-friction workaround available inside the current app constraints.
2. If the best practical path is:
   - show/copy the Goodreads export URL
   - instruct the user to paste it into Safari
   - or open a browser-safe intermediary page
   do that.
3. Choose the smoothest realistic option without overengineering.
4. Keep the change practical and shippable.

F. Keep the UI calm and premium
Requirements:
1. Do not turn this into a wall of technical troubleshooting.
2. Use short, explicit copy and strong hierarchy.
3. The screen should still feel like part of the product, not a support article.

G. Preserve all current functionality
Do not break:
- Open Goodreads Export Page CTA behavior unless intentionally improving it
- Choose CSV File
- web/mobile branching
- parsing/staging/execution flow

Implementation guidance
1. This is primarily an instructional-UX and flow-environment fix.
2. The central issue is app hijacking into Goodreads, not import logic.
3. Prefer the smoothest realistic browser-based workaround.
4. Make the happy path obvious and the failure path recoverable.
5. Keep the code change surgical.
