## 1. Why P2C exists (gap analysis vs P2A's partial coverage)

P2A shipped soft-avoid handling at retrieval, but only partially. Survey of
current touchpoints (lib/retrieval/, lib/recommender.ts, lib/recPolicy.ts):

| Surface                                          | Soft-avoid behavior today                                                              | Status |
|--------------------------------------------------|----------------------------------------------------------------------------------------|--------|
| statedGenres branch                              | Drops favorites that ALSO appear in soft-avoids (defense-in-depth)                     | OK     |
| revealedLanes branch — dense, per-anchor         | Skips lanes whose `LANE_TO_AFFINITY` maps to a soft-avoided AffinityKey                | OK     |
| revealedLanes branch — sparse, per-anchor        | Skips topGenres that are themselves in soft-avoids                                     | OK     |
| revealedLanes branch — branch-level quota cut    | Multiplies quota by `SOFT_AVOID_RETRIEVAL_MULTIPLIER=0.5` ONLY when **dominant lanes** intersect — sparse users (no dominant lanes) never trigger this | **GAP A** |
| revealedAuthors branch                           | Zero soft-avoid handling                                                                | **GAP B** (acknowledged limit, see §3) |
| `liked_subjects` fallback inside revealedLanes   | No mask — free-form subject strings vs AffinityKeys not comparable today               | **GAP C** |
| Catalog/local fallback (`local:fallback_scan` @ recommender.ts:~2640) | No soft-avoid filter on the SQL `.select()`                                            | **GAP D** |
| `getLocalCandidates` (`local:eligible`)          | No soft-avoid filter                                                                    | **GAP D** |
| Cross-branch trace surfacing                     | `RetrievalPlan.softAvoidLanesApplied` exists; `FetchItem.softAvoidDeprioritized?` field exists in `lib/retrieval/types.ts:47` but is never set | **GAP E** |
| `RankedRecsResult.meta`                          | No `soft_avoid_*` aggregate counter for debug surfaces                                  | **GAP E** |

P2C closes Gaps A, C, D, E and **explicitly defers Gap B** with a documented
rationale (see §3).

## 2. Scope (what P2C ships)

### S1. Sparse-user branch-quota trigger (Gap A)
- Extend `softAvoidedLanes()` semantics — or add a sibling helper
  `softAvoidedTopGenres()` — so the planner can detect intersections in BOTH
  `ctx.dominantLanes` (dense) AND `ctx.topGenres` (sparse).
- `lib/retrieval/branchPlanner.ts`: soft-avoid quota multiplier (`SOFT_AVOID_RETRIEVAL_MULTIPLIER`)
  fires whenever EITHER intersection is non-empty. `softAvoidLanesApplied` trace
  field continues to record the AffinityKeys that triggered the reduction.
- No change to `BRANCH_QUOTAS` constants.
- No change to multiplier value (`0.5`) — calibration knob, owned by `recPolicy.ts`.

### S2. `liked_subjects` fallback mask (Gap C — bounded)
- Inside `buildRevealedLanesBranch` sparse path, the `liked_subjects` loop
  consults a new `lib/recPolicy.ts` constant `LIKED_SUBJECT_AVOID_GUARDS:
  Partial<Record<AffinityKey, readonly string[]>>` — a small curated list of
  subject substrings that obviously belong to each AffinityKey (e.g.
  `fantasy_scifi → ['fantasy', 'sci-fi', 'science fiction', 'dragons',
  'wizard']`). If a `liked_subject` matches a guard for a soft-avoided
  AffinityKey, it is skipped.
- Deliberately a guard list, NOT a free-text classifier — stays auditable and
  removes the LLM/embeddings dependency that would otherwise leak into a free-
  plane surface.
- Empty-by-default safe: if the constant is empty, behavior is identical to
  pre-P2C.

### S3. Catalog/local soft-avoid filter (Gap D)
- `lib/recommender.ts` `getLocalCandidates` (and the fallback `.select()` block
  near line 2640): when `req?.signals.softAvoids.genres.length > 0`, post-filter
  the local candidate list to demote — NOT exclude — books whose `primaryGenre`
  ∈ softAvoids.
- "Demote" semantics: keep at most `Math.floor(softAvoidedCount * (1 -
  SOFT_AVOID_RETRIEVAL_MULTIPLIER))` of them, sampled by the existing local
  ordering (no shuffle — determinism preserved). This mirrors the lane-quota
  multiplier shape rather than introducing a new "exclude" semantic.
- If `req` is absent (legacy callers), no-op — preserves pre-P2A retrieval
  shape exactly.

### S4. Per-`FetchItem` and `RankedRecsResult.meta` trace surfacing (Gap E)
- `lib/retrieval/types.ts`: keep `FetchItem.softAvoidDeprioritized?: boolean`;
  populate it (`true`) on revealedLanes items emitted under reduced quota AND
  on revealedLanes items emitted from a per-anchor mask near-miss
  (informational — does not change retrieval).
- `lib/recommender.ts` extend `RankedRecsResult.meta` with optional
  `soft_avoid_retrieval?: { lanesApplied: AffinityKey[]; localDemoted: number;
  branchQuotaMultiplier: number }` populated from the planner result + local
  filter delta. Optional field — debug surfaces can read it; existing readers
  compile unchanged.

### S5. Validator extension
- `scripts/validate_retrieval_planner.ts` adds A7-class cases:
  - **A7a** sparse user with topGenres ∩ softAvoids ≠ ∅ → revealedLanes quota
    reduced; previously A7 only covered dense.
  - **A7b** sparse user with NO topGenres intersection → no quota reduction
    (regression guard).
  - **A7c** `liked_subject` matching `LIKED_SUBJECT_AVOID_GUARDS` for a soft-
    avoided AffinityKey is skipped; non-matching subjects pass through.
  - **A7d** local-candidate post-filter respects `SOFT_AVOID_RETRIEVAL_MULTIPLIER`
    (input N candidates with primaryGenre ∈ softAvoids → output ≈ N/2).
  - **A7e** legacy caller (`req=undefined`) → no soft-avoid filtering applied
    (legacy parity).
- No live OL / Supabase / DB calls — same shape as the existing validator.

## 3. Out-of-scope (explicit rejections, with rationale)

### Gap B — revealedAuthors branch soft-avoid
**Deferred to P4 (hard-avoid global pre-branch exclusion) or later.** An author
whose dominant work is in a soft-avoided genre would require an author→genre
mapping the codebase doesn't have today. Implementing it in P2C would either
(a) require a new lookup table (adds maintenance surface for marginal value;
soft-avoid is "less of, not none of"), or (b) require post-fetch per-book
filtering on retrieved-by-author candidates (acceptable as a downstream filter
but conflates the per-anchor branch responsibility with cross-cut filtering —
which is the P4 hard-avoid shape). Document the limitation in `replit.md`
gotchas; do not paper over.

### Hard candidate-level exclusion at retrieval
Soft avoid means **less of, not exclude**. A hard pre-branch exclusion is
explicitly P4 territory and would silently widen the soft/hard distinction
the architecture spec keeps separate.

### `RecRequest` / `BuildCause` / signal-class surface changes
None. P2C consumes the existing `req.signals.softAvoids.genres` array and
the existing `BuildCause` enum unchanged.

### Composition / scoring / explanation faithfulness
None. P2C is retrieval-side only. The composition layer's P2B + P2B.1
reservation logic and the P1 stated_taste contribution math are untouched —
P2C reduces upstream supply; downstream demand handling is unchanged.

## 5. Soundness checks vs P2B.1 contract

P2B.1 reserves `pickStatedReservation` to candidates that satisfy BOTH
`book._retrieval_reason.startsWith('stated_genre:')` AND
`_score_breakdown.stated_taste > 0`. P2C changes:

- **statedGenres branch unchanged** — emits `stated_genre:<key>` reasons
  identically. P2B.1 retrieval-provenance check passes for any book the
  `statedGenres` branch returned.
- **No multi-source `_retrieval_reasons[]` introduction** — first-seen-wins
  dedup still wins; P2B.1's reliance on the prefix string is preserved.
- **No new branch added** — `BRANCH_ORDER` is still
  `statedGenres → revealedAuthors → revealedLanes`, so any cross-branch
  duplicate book retains the earliest branch's `_retrieval_reason`.
- **Local soft-avoid post-filter operates on `local:*` candidates only** —
  these books carry `_retrieval_reason='local:eligible'` /
  `'local:fallback_scan'` and correctly fail the P2B.1 retrieval gate. The
  filter cannot promote a non-stated book into a stated-eligible position.

## 7. Known limitation P2C does NOT close

- Gap B (revealedAuthors soft-avoid) — see §3.
- End-to-end deck-shift verification on a live account (the V1–V4 caveat in
  `replit.md` already flags this as outstanding for the broader stream).
- Subject-string ↔ AffinityKey mapping for arbitrary `liked_subjects` is
  guard-list bounded (`LIKED_SUBJECT_AVOID_GUARDS`) — not a full classifier.
  Subjects outside the guard list pass through unfiltered. Acceptable: soft
  avoid is bias not exclusion, and the guard list is editable.
