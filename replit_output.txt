Q2. Are adjacent_fit stated candidates currently ineligible?
----------------------------------------------------------
YES — categorically.

Exact rule: lib/composition/statedReservation.ts:127-128

   const isCore = sb.fit_class === 'core_fit' && !sb.audit_flags?.includes('weak_metadata');
   if (!isCore && !STATED_RESERVATION_POLICY.allowAdjacentReservation) continue;

Was it intentional? YES at the policy level. lib/recPolicy.ts:240-242
documents the choice:

   /** When false, only CORE candidates may be reserved. Conservative default —
    *  setting true would silently widen ADJACENT exposure on every pref edit. */
   allowAdjacentReservation:  boolean;
   …
   allowAdjacentReservation: false,

Intent was conservatism (don't dilute slate quality). It was NOT designed
to be permanent — `allowAdjacentReservation` is a tunable policy knob, not
a hard architectural rule. The header at lib/recPolicy.ts:227 explicitly
calls these "calibration hypotheses; tunable without architectural change."

Conflict with the Phase 2 product contract? YES, in this exact failure
mode. The pre-beta gate says "visible deck shift after pref edit, dense
users included." For dense users an explicit edit toward an off-lane genre
will routinely produce adjacent_fit (not core_fit) candidates, because
core_fit requires alignment with the user's REVEALED dominant lane. Holding
reservation to core-only means the deck shift gate cannot pass for the
exact population it was written to protect.

Q4. Did P2B/P2B.1 validators miss this case?
----------------------------------------------------------
YES — the live failure class is not exercised by the validator's product-
behavior assertions.

scripts/validate_stated_reservation.ts (cases C1-C13) enumerated:
  C1-C5    happy-path reservation, all candidates fitClass='core_fit'
  C6       "Only candidate is ADJACENT and policy disallows → null"
            asserts pick === null AND
            STATED_RESERVATION_POLICY.allowAdjacentReservation === false
  C7-C13   weak_metadata / sort-order / cap-respect / etc., all fitClass='core_fit'

Coverage analysis vs. live failure:

  ❌ NOT covered: an "all candidates ADJACENT, zero CORE" pool produced by
                  realistic post-edit retrieval for a dense user.
  ⚠️ Inverted:    C6 PROVES the current behavior is mechanically correct
                  (returns null) — but it asserts that as the DESIRED
                  behavior, treating ADJACENT-only as a non-product case.
                  No validator case asks "did the pref edit visibly shift
                  the deck?" because that is a product-acceptance question,
                  not a unit invariant.

Net: validators prove the helper's contract (CORE-only, single-slot, caps
respected). They cannot — and were never designed to — fail when the
product-acceptance promise fails. The mechanical contract is being met;
the product promise is not.

There is no validator gap to "fix" by adding a unit case. The gap is the
absence of a product-level acceptance fixture (e.g., dense-user-edit-to-
off-lane-genre snapshot replaying through composition and asserting at
least one stated_genre:* book lands in top-4). That fixture would have
caught this at CI time.

Q5. Is the current rule too weak for Phase 2 acceptance?
----------------------------------------------------------
YES — and this run is the canonical proof.

The acceptance gate ("visible deck shift after pref edit, dense users
included") commits us to two things:
  (i)  retrieval makes stated candidates available, and
  (ii) at least one of them surfaces in the visible top slate.

For dense users an explicit edit toward a genre off their revealed
dominant lane is the COMMON case (the whole point of editing prefs is
usually to nudge AWAY from prior behavior). computeFitClass will
classify those candidates as adjacent_fit by design — that's the
classifier doing its job.

If reservation is core-only, then:
  • a dense user editing toward an off-lane genre never sees the deck shift,
  • a dense user editing toward an on-lane genre would already see it
    without any reservation needed (those candidates would land in the
    top slate organically),
  • therefore reservation as currently configured is a no-op for the
    population it was written to serve.

That is "too weak" against the stated contract. The product promise of
P2 cannot be met under the current policy.

Caveat: widening to adjacent unconditionally could meaningfully lower
top-4 quality for users whose edits don't actually shift their genre
mix. The mitigation lever is the OTHER three gates (provenance AND
scoring-flag AND composition caps) — those still protect against
spurious reservations even when fit_class is widened.
