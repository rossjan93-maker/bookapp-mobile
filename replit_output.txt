2. PROPOSED CONTRIBUTION MODEL
================================================================================
The smallest sound shape that supports BOTH free explanation faithfulness now AND
premium semantic explanations later (P4+ pluggability):

```ts
// lib/scoring/contributions.ts (NEW)
export type ContributionSource =
  | 'stated_durable'         // SignalClass — from RecRequest stated signals
  | 'revealed_behavioral'    // SignalClass — from rated history
  | 'current_intent'         // SignalClass — from active intent chip
  | 'short_term_feedback'    // SignalClass — MLT / "more like this"
  | 'soft_avoid'             // SignalClass — penalty-only
  | 'retrieval_provenance'   // P2A/P2B.1 retrieval branch
  | 'quality_reliability'    // hygiene / metadata floor
  | 'novelty_diversity'      // future P4 hook (disabled-by-default in free)
  | 'semantic_rerank';       // future P4 hook (disabled-by-default in free)

export type ContributionKind =
  | 'trait_alignment'
  | 'appreciation_match'
  | 'theme_overlap'
  | 'genre_affinity'
  | 'feedback_boost'
  | 'reader_consensus'
  | 'stated_taste_fit'      // P1 STATED_TASTE_POLICY scoring path
  | 'intent_fit'
  | 'soft_avoid_penalty'
  | 'retrieval_anchor'      // book was surfaced because branch X retrieved it
  | 'cog_fit'               // center-of-gravity adjustment
  | 'hygiene_floor';

export interface ScoreContribution {
  source: ContributionSource;
  kind: ContributionKind;
  value: number;            // signed; negatives are penalties
  evidence: string[];       // raw tokens (genre keys, trait keys, branch prefix)
  explanationEligible: boolean; // false for hygiene_floor, soft_avoid_penalty,
                                // true for everything user-facing-positive
}
```

Rationale for these choices:
- `source` mirrors the SignalClass taxonomy already living in `lib/recSignals/`
  (build.ts L67/L75/L80/L87/L92) PLUS two engine-side sources
  (`retrieval_provenance`, `quality_reliability`) PLUS two reserved P4 hooks.
  No new vocabulary invented.
- `kind` enumerates the existing reason families already present in
  recommender.ts (L1527, L1529, L1566, L1592, L1614, L1655). Mapping is 1:1
  with what scoring already does — no new behavior.
- `evidence: string[]` is the bridge that lets the composer cite specifics
  ("aligns with literary, character-driven") without touching scoring math.
- `explanationEligible` makes hygiene/penalty contributions invisible to the
  composer without filtering them out of the audit trail.
- `value` stays a single number — keeps math compatible with existing
  `ScoreBreakdown`. P3 does NOT change weights, caps, or floors.

Attachment site: `ScoredBook` gains
```ts
contributions?: ScoreContribution[];   // NEW — primary truth for explanations
reasons: string[];                     // EXISTING — derived, presentation-only
```

`ScoreBreakdown` (recommender.ts:154) stays as-is. It is the additive math
view; `contributions[]` is the auditable evidence view. Both are derived from
the same scoring pass.

5. P2 PROVENANCE — TWO RESIDUALS RE-EVALUATED
================================================================================
The mapping question explicitly asks whether each P2B.1 residual belongs in P3.

Residual #1: `fetchOLByAuthor` hardcodes `author_anchor:${author}` (recommender.ts:715)
  Architectural impact on P3: MEDIUM.
  - Without fixing: an author-branch contribution will use a `_retrieval_reason`
    that does not match the prefix taxonomy used by `pickStatedReservation` or
    by the future composer dispatch table. Composer would need a special-case.
  - Cost to fix: ~5 lines + one new prefix constant.
  - Recommendation: **FIX IN P3.** Cheap, eliminates a special case in the
    composer dispatch.

Residual #2: merge-dedup drops loser-branch provenance (recommender.ts:570-600)
  Architectural impact on P3: HIGH.
  - Without fixing: a book retrieved by BOTH `stated_genres:fantasy_scifi` AND
    `revealed_lanes:literary` keeps only the last writer's `_retrieval_reason`.
    The composer will pick whichever branch happened to write last, not the
    strongest truthful contribution.
  - This DIRECTLY affects A8 (zero false-attribution) and A9 (slate diversity):
    if every dense user's book ends up tagged `revealed_lanes:` because that
    branch runs last, the slate cannot diversify on retrieval source.
  - Cost to fix: extend `CandidateBook` with `_retrieval_reasons: string[]` and
    have the merge step append rather than overwrite. Existing
    `_retrieval_reason` becomes a derived "primary" (first entry). Backwards
    compatible.
  - Recommendation: **FIX IN P3.** Required for A8/A9 to be provable.

Both residuals upgraded from "P2 deferred" to "P3 prerequisite."

P2C `softAvoidDeprioritized` flag (FetchItem in `lib/retrieval/types.ts:47`):
  Already propagates to ScoredBook through the existing pipe. P3 reads it as
  evidence on the `soft_avoid_penalty` contribution. No new plumbing needed.

P1 BuildCause: already propagates via `consumePendingBuildCause`
(recRequest.ts:78) and `buildRecRequest` (L180). P3 quality gate A10 verifies

6. SLATE-LEVEL DIVERSITY
================================================================================
Architecture decision: minimal version IN P3, full version deferred to P4.

Minimal P3 pass (in `lib/explanations/compose.ts`):
- After per-book composition, walk top-N (default N=4).
- For each card, the composer has produced a ranked list of eligible
  ContributionKinds that survived the per-kind floor.
- If ≥3 of top-4 share the same primary kind AND any has a viable secondary
  contribution above floor, swap that card's primary to its secondary.
- Never invent a reason. Never lower a per-kind floor for diversity.
- Determinism: tiebreak by ContributionKind enum order.

What this is NOT: P3 does NOT add a "global slate optimizer," does NOT
re-rank books for diversity, does NOT change top-slate reservation.
Reservation winners keep their slot; only the displayed reason text may
swap to a different truthful contribution.

Risk: low. The pass is read-only over scoring output and only changes which
of the existing eligible contributions becomes the headline.

8. MIGRATION STRATEGY (reasons[] → contributions)
================================================================================
Two-step transition within a single PR — no flag, no deprecation cycle:

Step A: contributions[] becomes the source of truth.
- Scoring emits contributions alongside the existing breakdown.
- Composer computes the user-facing explanation string from contributions.
- `book.reasons` is repopulated from the composer output (so RecCard's
  fallback path produces the same text the composer would).

Step B: legacy `reasons.push(...)` sites are replaced with contribution
        emissions.
- Hygiene/penalty `reasons.push` calls become non-eligible contributions
  (kept for forensic logging, never displayed).
- "LOOSE" trait/genre/feedback/consensus pushes are deleted; their
  contributions emit during scoring.
- The old `reasons[0] = CoG fit_explanation` overwrite at L1993-1996 is
  removed; CoG becomes a `cog_fit` contribution with `value = cog_score_delta`
  and `evidence = [book.fit_classification]`.

`book.reasons` survives as a presentation/compatibility output that any
downstream consumer (RecCard, audit scripts, dev inspector) can still read.

No external API or persisted shape changes. AsyncStorage / Supabase rows
unaffected.

9. VALIDATION STRATEGY
================================================================================
New validator: `scripts/validate_explanation_faithfulness.ts`

Quality gates the validator must prove (deterministically, no live network):

A8 — Zero false-attribution (HARD GATE):
  - Synthesize 20+ fixture users × N candidates with known input signals.
  - Run the composer. For every emitted reason, assert there exists a
    `ScoreContribution` with matching `kind` AND `value >= MIN_DISPLAY[kind]`
    AND `explanationEligible === true`.
  - Counter-fixtures: book with NO matching contribution must produce NO
    reason of that kind.
  - Stated-avoid users must produce no "you've been reading X" reason.

A9 — Slate diversity (SOFT GATE):
  - 6+ fixture top-slate compositions where ≥3 books share the same primary
    contribution kind AND have viable secondaries.
  - Assert diversity guard swaps where possible.
  - Assert NO swap when no viable secondary exists (truth wins over diversity).

A10 — BuildCause propagation (HARD GATE):
  - Construct `RecRequest` with each `BuildCause` value.
  - Assert `RetrievalTrace.build_cause` matches; assert
    `explicit_preference_edit` triggers reservation; assert
    `onboarding_completion` does not.

Provenance distinction:
  - Stated-genre reason text uses stated-template copy, never revealed-template.
  - Revealed-history reason text never claims anchor for a stated-only signal.
  - Intent reason fires iff `intent_fit.value > 0`.
  - Feedback reason fires iff `feedback_boost.value > 0`.

Existing validators continue to pass:
- `scripts/validate_taxonomy.ts`
- `scripts/validate_rec_validity.ts`
- `scripts/validate_retrieval_planner.ts` (75 cases)
- `scripts/validate_stated_reservation.ts` (56 cases)
- `scripts/explanation_audit.ts` (extended, not replaced)

11. DECISIONS REQUIRING HUMAN APPROVAL BEFORE CODE
================================================================================

D1 — Promote multi-source retrieval provenance into P3 (`_retrieval_reasons: string[]`)
  Recommendation: **APPROVE.**
  Justification: Without it, A8/A9 are not provable for books retrieved by
  multiple branches. The merge-dedup overwrite is the dominant source of
  silent provenance loss in dense users. Cost is small (one type field +
  append-instead-of-overwrite at the merge site). The P2B.1 AND-gate keeps
  identical semantics — it just consults the array instead of the scalar.

D2 — Fix `fetchOLByAuthor` hardcoded retrieval reason label in P3
  Recommendation: **APPROVE.**
  Justification: Cheap (~5 lines + one prefix constant). Removes a special
  case from the composer dispatch table. Author-branch contributions become
  first-class citizens of the prefix taxonomy.

D3 — Slate diversity in `lib/explanations/diversityGuard.ts` as separate file
  Recommendation: **APPROVE separate file** (vs folding into compose.ts).
  Justification: Easier to unit-test in isolation; future P4 enhancements
  (premium evidence-cited explanations) plug in here without disturbing the
  per-book composer.

D4 — Per-kind display floors live in `lib/scoring/contributions.ts`
       (NOT `lib/recPolicy.ts`)
  Recommendation: **APPROVE.**
  Justification: These are explanation-display thresholds, not scoring policy.
  They control what is shown to users, not what wins ranking. Keeping them
  separate from `recPolicy.ts` preserves the principle that recPolicy owns
  scoring-side calibration only.

D5 — `book.reasons[]` is preserved as compatibility output, NOT removed
  Recommendation: **APPROVE.**
  Justification: Keeps RecCard's fallback path working; keeps existing
  audit scripts compatible; allows a future cleanup PR to remove it once
  every consumer is verified contribution-aware.

No other decisions block implementation.

12. RISKS & MITIGATIONS
================================================================================
R1: Composer produces empty explanation for a book with all-below-floor
    contributions.
  Mitigation: Fall back to a non-personalized template ("New for you") with
  `source = 'quality_reliability'` so the card still has copy. Gate this on
  A8 — non-personalized copy must not claim history.

R2: Diversity guard suppresses the strongest truthful reason for a card.
  Mitigation: Guard never demotes a contribution if the swap would drop
  `value` below 70% of the primary's value. Determined in §6 spec.

R3: Multi-source provenance expansion breaks `pickStatedReservation`.
  Mitigation: AND-gate becomes "any element of `_retrieval_reasons`
  startsWith('stated_genre:')" — strict superset of current behavior. Every
  existing test fixture continues to satisfy it.

R4: Performance — adding contributions[] per book on a 200-candidate scoring
    pass.
  Mitigation: Contributions are typed structs already computed inline;
  emission is one push per existing component evaluation. No new loops.
  Estimated overhead: <2% of scoring pass.

R5: Validator fixture maintenance burden grows.
  Mitigation: New validator builds on the existing planner/reservation
  fixture style; one shared fixture builder helper in the validator file.
